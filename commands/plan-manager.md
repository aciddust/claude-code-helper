# 계획 관리 스킬

> 모든 개발 작업은 **계획 수립 → 승인 → 실행** 순서를 따른다.
> 이 스킬은 계획 수립 단계를 자동화한다.

## 실행 흐름

> ⚠️ **경로 규칙**: `/plan-manager` 시작 시 가장 먼저 Bash 도구로 `pwd`를 실행하여
> 현재 절대 경로(`PROJECT_ROOT`)를 확인한다. 이후 **모든 파일 생성·수정은 이 경로를 기준**으로 하며,
> 상위 디렉터리에 파일을 만들지 않는다.

```
1. 사용자가 작업 지시
       │
       ▼
2. 계획 수립 (이 스킬)
   ├── 요구사항 분석
   ├── 기술 조사
   └── 구현 전략 설계
       │
       ▼
3. 3문서 생성 → docs/plans/{작업명}/
   ├── PLAN.md         계획서
   ├── CONTEXT.md      맥락 노트
   └── CHECKLIST.md    체크리스트 (상태: 🔴 시작 전)
       │
       ▼
4. ⛔ 여기서 반드시 멈춘다
   ├── 계획 요약을 사용자에게 보여준다
   ├── "검토해주세요" 라고 요청한다
   └── 사용자 응답이 올 때까지 코드를 작성하지 않는다
       │
       ▼
5. 사용자 승인
   ├── 승인 시 → CHECKLIST.md 상태를 🟡 진행 중 으로 변경 → 구현 시작
   └── 수정 요청 시 → 3문서 수정 → 다시 4번으로
```

## 3문서 작성 규칙

### 문서 1: PLAN.md (계획서)

**목적**: 뭘 만들 건지, 처음부터 끝까지

```markdown
# [작업명] 계획서

## 개요
- 작업 목적 (한 줄 요약)
- 예상 범위 (영향받는 파일/모듈)
- 예상 소요 (단계 수)

## 현재 상태 분석
- 기존 코드 구조
- 변경이 필요한 부분

## 구현 계획
### Phase 1: [단계명]
- 구체적 작업 내용
- 예상 변경 파일

### Phase 2: [단계명]
...

## 기술 선택
- 사용할 라이브러리/패턴
- 선택 이유 (간략)

## 리스크
- 예상되는 문제점
- 대응 방안
```

### 문서 2: CONTEXT.md (맥락 노트)

**목적**: 왜 이렇게 결정했는지, 자료 위치

```markdown
# [작업명] 맥락 노트

## 결정 기록
| 결정 사항 | 선택지 | 최종 선택 | 이유 |
|-----------|--------|-----------|------|

## 참조 자료
- 관련 문서/URL
- 참고한 코드 위치

## 주요 제약 조건
- 기술적 제약
- 비즈니스 제약

## 사용자 요구사항 원문
> (사용자가 지시한 내용 그대로 기록)
```

### 문서 3: CHECKLIST.md (체크리스트)

**목적**: 뭘 끝냈고, 뭐가 남았는지 추적

```markdown
# [작업명] 체크리스트

## 진행 상태: 🔴 시작 전 | 🟡 진행 중 | 🟢 완료

## 작업 목록
- [ ] Phase 1: [단계명]
  - [ ] 세부 작업 1
  - [ ] 세부 작업 2
- [ ] Phase 2: [단계명]
  - [ ] 세부 작업 1

## 컨텍스트 전환 체크
- [ ] 사용자 승인 완료
- [ ] /clear 안내 출력 완료
- [ ] /clear 후 복귀 또는 "계속 진행" 확인

## 품질 체크
- [ ] 에러 처리 적용
- [ ] 보안 검토 완료
- [ ] 테스트 작성/통과
- [ ] 코드 리뷰 포인트 확인

## 변경 로그
| 시간 | 파일 | 변경 내용 | 상태 |
|------|------|-----------|------|
```

## 문서 저장 위치

```
$PROJECT_ROOT/docs/plans/{작업명}/
├── PLAN.md
├── CONTEXT.md
└── CHECKLIST.md
```

- `PROJECT_ROOT`는 `/plan-manager` 호출 시 `pwd`로 확인한 절대 경로다
- `{작업명}`은 kebab-case로 (예: `user-auth`, `api-refactor`)
- 한 작업 = 한 폴더, 문서 3개가 항상 세트

## 승인 대기 규칙

3문서를 생성한 후, 반드시 다음 형식으로 사용자에게 요약을 보여주고 멈춘다:

```
📋 계획 수립 완료 — 검토를 요청합니다

📂 docs/plans/{작업명}/
  ├── PLAN.md       ← 전체 구현 계획
  ├── CONTEXT.md    ← 결정 근거
  └── CHECKLIST.md  ← 작업 체크리스트 (🔴 시작 전)

[계획 요약]
  Phase 1: {단계1}
  Phase 2: {단계2}
  ...

⏸️ 검토 후 승인해주세요. 승인 전까지 코드 작성을 시작하지 않습니다.
   수정할 부분이 있으면 말씀해주세요.
```

**⛔ 절대 금지: 이 메시지 출력 후 같은 턴에서 다음 행위를 하지 않는다:**
- 코드 파일을 읽거나(Read) 수정(Edit/Write)하지 않는다
- Bash 명령을 실행하지 않는다
- Task 도구로 서브에이전트를 호출하지 않는다
- 유일하게 허용되는 것: 사용자에게 텍스트로 응답하는 것
**사용자의 다음 메시지가 올 때까지 반드시 대기한다.**

사용자가 "확인", "승인", "진행", "좋아", "ㅇㅇ" 등 동의 의사를 표현하면:
1. CHECKLIST.md의 상태를 `🔴 시작 전` → `🟡 진행 중` 으로 변경한다
2. **컨텍스트 전환 절차**를 실행한다 (아래 참조)
3. 그 다음 턴부터 구현을 시작한다

## 승인 후 컨텍스트 전환 절차

사용자가 계획을 승인하면, **반드시 아래 순서를 실행한다**. 이 절차는 계획 수립 과정에서 소모된 컨텍스트를 정리하고, 구현에 필요한 정보만으로 리셋하는 것이 목적이다.

### Step 1: CHECKLIST.md 상태 변경 및 승인 체크
```
🔴 시작 전 → 🟡 진행 중
```
동시에 "컨텍스트 전환 체크" 섹션의 첫 번째 항목을 체크한다:
```
- [x] 사용자 승인 완료
```

### Step 2: 컨텍스트 초기화 안내 후 멈춤

상태 변경 직후, **아래 메시지만 출력하고 반드시 멈춘다.** 구현을 시작하지 않는다. 파일을 읽지 않는다. 코드를 작성하지 않는다.
이 메시지를 출력할 때 "컨텍스트 전환 체크" 섹션의 두 번째 항목을 체크한다:
```
- [x] /clear 안내 출력 완료
```

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 계획이 승인되었습니다
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📂 docs/plans/{작업명}/CHECKLIST.md → 🟡 진행 중

계획 수립 과정에서 컨텍스트가 많이 사용되었습니다.
깨끗한 상태에서 구현을 시작하면 더 원활합니다.

👉 /clear 를 실행한 뒤 "이어서 구현해줘" 라고 입력하세요.
   plan-guard 훅이 진행 상황을 자동으로 안내합니다.

(바로 진행하려면 "계속 진행" 이라고 입력하세요)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**⛔ 절대 금지: 이 메시지를 출력한 후 이 턴에서 어떤 도구도 호출하지 않는다.**
- CHECKLIST.md 상태를 이미 변경했으므로 Write/Edit 필요 없음
- 계획 문서를 다시 읽을 필요 없음 (다음 턴에서 읽는다)
- Step 3~6은 **반드시 사용자의 다음 메시지 이후에** 실행한다

사용자가 `/clear` 후 다시 돌아오면 → plan-guard 훅이 🟡 진행 중을 감지하여 진행 상황을 자동 주입한다.
사용자가 "계속", "진행", "넘어가" 등 응답하면 → Step 3으로 진행한다.

### Step 3: 계획 문서를 파일에서 다시 읽기
"컨텍스트 전환 체크" 섹션의 세 번째 항목을 체크한다:
```
- [x] /clear 후 복귀 또는 "계속 진행" 확인
```
대화 히스토리의 내용에 의존하지 않고, **반드시 파일을 직접 Read 도구로 다시 읽는다**:
- `docs/plans/{작업명}/PLAN.md` — 전체 구현 계획
- `docs/plans/{작업명}/CHECKLIST.md` — 작업 목록

### Step 4: 매뉴얼 챕터 참조
작업 유형에 맞는 `/dev-manual` 챕터를 읽는다.

### Step 5: 구현 시작 메시지 출력
```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
🚀 구현을 시작합니다
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

📂 작업: {작업명}
📄 계획: docs/plans/{작업명}/PLAN.md

현재 Phase: Phase 1 — {단계명}
  - [ ] {세부작업1}
  - [ ] {세부작업2}

전체 진행: 0/{총 작업 수} 완료

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### Step 6: Phase 1부터 구현 시작
- CHECKLIST.md의 첫 번째 미체크 Phase부터 순서대로 구현한다
- 한 Phase가 끝나면 CHECKLIST.md를 업데이트하고 다음 Phase로 넘어간다

## 중요 규칙

1. **계획 먼저**: 코드 한 줄 쓰기 전에 3문서부터 생성
2. **승인 필수**: 문서 생성 후 반드시 사용자에게 검토 요청. **같은 턴에서 코드 작성 금지**
3. **상태로 추적**: `🔴 시작 전` = 미승인, `🟡 진행 중` = 승인됨, `🟢 완료` = 작업 종료
4. **실시간 갱신**: 작업 중 CHECKLIST.md는 계속 업데이트
5. **맥락 보존**: 새 세션에서도 docs/plans/ 읽으면 이어서 작업 가능
