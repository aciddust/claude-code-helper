# 프로젝트 설정 매니저

> **`/setup` 하나로 모든 것을 설정한다**
> 프로젝트 비전부터 첫 번째 개발 계획까지, 5단계 대화형 매니저로 진행한다.

---

## 핵심 원칙

1. **반드시 단계별로 진행한다** — 한 Phase의 질문에 대한 답변을 받은 후에만 다음 Phase로 넘어간다
2. **한 번에 모든 질문을 쏟아붓지 않는다** — Phase별로 끊어서 질문하고, 사용자 확인을 받는다
3. **기존 코드가 있으면 자동 분석을 먼저 한다** — 사용자에게 물어보기 전에 감지 가능한 건 먼저 분석
4. **모든 단계에서 사용자 확인을 받는다** — "이렇게 진행할까요?" 확인 후 다음 단계

## Phase 1: 프로젝트 비전 수집

> **목표**: 이 프로젝트가 뭘 하는 건지, 어디까지 만들 건지 파악

### 최우선: 작업 디렉터리 고정

**Phase 1 시작 전 가장 먼저** Bash 도구로 현재 절대 경로를 확인한다:

```bash
pwd
```

출력된 경로가 이 세션의 `PROJECT_ROOT`다. **이후 모든 파일 생성·수정·경로 참조는 반드시 이 절대 경로를 기준으로 한다.** 상위 디렉터리나 다른 위치에 파일을 만들지 않는다. Write/Edit 도구를 사용할 때는 항상 `PROJECT_ROOT`를 앞에 붙인 절대 경로를 사용한다.

### 시작 전 준비

Phase 1 시작 메시지를 출력하기 **직전에** 반드시 `.setup-in-progress` 마커를 생성한다:

```bash
PROJECT_ROOT=$(pwd) && mkdir -p "$PROJECT_ROOT/.claude" && touch "$PROJECT_ROOT/.claude/.setup-in-progress"
```

이 마커가 있어야 setup 진행 중 사용자의 답변이 plan-guard 훅에 차단되지 않는다.

### 시작 메시지

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
프로젝트 설정 매니저를 시작합니다
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

5단계를 거쳐 개발 환경을 완벽하게 세팅합니다:

  Phase 1  프로젝트 비전        ← 지금
  Phase 2  기술 환경
  Phase 3  워크플로우 설정
  Phase 4  초기 개발 계획
  Phase 5  환경 세팅 적용

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

### 질문 그룹 1-A: 프로젝트 개요

사용자에게 다음을 질문한다. **한 번에 전부 물어본다:**

```
Phase 1: 프로젝트에 대해 알려주세요

1. 프로젝트 이름은 무엇인가요?
2. 이 프로젝트를 한 문장으로 설명하면?
   (예: "SNS 컨셉의 음악 스트리밍 서비스")
3. 현재 상태는?
   - [ ] 아이디어 단계 (코드 없음)
   - [ ] 기존 프로젝트에 이 시스템 적용
4. 핵심 기능을 3~5개 나열해주세요
   (예: "사용자 인증, 대시보드, 실시간 알림")
5. 첫 번째로 만들고 싶은 기능은?
   (이 기능으로 초기 개발 계획을 세웁니다)
```

### 답변 처리

- 사용자 답변을 정리하여 내부적으로 `PROJECT_VISION`으로 저장 (파일에 쓰지 않고 기억)
- 답변이 불충분하면 추가 질문 (예: "실시간 알림은 WebSocket 기반인가요? Push 알림인가요?")
- 충분하면 확인:

```
프로젝트 비전 확인:

  이름:       {프로젝트명}
  설명:       {한 줄 설명}
  상태:       {신규/기존}
  핵심 기능:   {기능 목록}
  첫 작업:     {첫 번째 기능}

이대로 진행할까요? (수정할 부분이 있으면 말씀해주세요)
```

**여기서 반드시 멈춘다.** 사용자가 "확인", "진행" 등 응답할 때까지 Phase 2로 넘어가지 않는다.

## Phase 2: 기술 환경 분석

> **목표**: 기술 스택, 프로젝트 구조, 개발 도구 파악

### 2-A: 자동 분석 (기존 프로젝트인 경우)

Phase 1에서 "기존 프로젝트"라고 답한 경우, **질문 전에 먼저 자동 분석**을 실행한다:

```
분석 항목:
├── package.json / pyproject.toml / go.mod / Cargo.toml → 기술 스택
├── 디렉토리 구조 (ls, tree) → 프로젝트 레이아웃
├── .eslintrc / prettier / ruff.toml / biome.json → 린터/포매터
├── tsconfig.json / jsconfig.json → 언어 설정
├── 기존 테스트 파일 → 테스트 프레임워크
├── .env / .env.example → 환경변수 패턴
├── Dockerfile / docker-compose → 배포 환경
├── 기존 코드 샘플 → 네이밍, import 순서 추론
├── CLAUDE_origin.md → 기존 프로젝트 설명/규칙 (install 시 백업된 원본)
└── docs_origin/ → 기존 문서/계획 (install 시 백업된 원본)
```

#### _origin 파일 활용

install.sh가 기존 `CLAUDE.md`와 `docs/`를 `CLAUDE_origin.md`, `docs_origin/`으로 백업한다.
이 파일들이 존재하면 **반드시 읽고 분석에 반영**한다:

- **`CLAUDE_origin.md`**: 기존 프로젝트의 설명, 코딩 규칙, 아키텍처 정보를 추출하여 Phase 2 분석 결과와 Phase 5의 CLAUDE.md 생성에 반영한다.
- **`docs_origin/`**: 기존 문서 구조, 계획, 가이드를 파악하여 프로젝트 맥락을 이해한다.

분석 결과를 정리하여 사용자에게 보여준다:

```
자동 분석 결과:

  언어:         TypeScript 5.9.3
  프레임워크:    Svelte 5.51.0 (SvelteKit)
  DB:           PostgreSQL + Prisma
  패키지 매니저: npm
  린터:         ESLint + Prettier
  테스트:        Vitest
  디렉토리:      src/app, src/components, src/lib, prisma/

맞나요? 틀린 부분만 수정해주세요.
(전부 맞으면 "확인" 이라고 답해주세요)
```

## Phase 3: 워크플로우 설정

## Phase 4: 초기 개발 계획 수립

> **목표**: Phase 1에서 "첫 번째로 만들고 싶은 기능"을 기반으로 초기 계획 수립

### 4-A: 첫 번째 기능 분석

Phase 1에서 받은 "첫 번째 기능"을 분석하고 개발 계획을 수립한다:

1. 기능 요구사항 분석
2. 필요한 파일/모듈 도출
3. 구현 단계(Phase) 설계
4. 리스크 식별

### 4-B: 3문서 생성

`docs/plans/{첫번째-기능-kebab-case}/` 에 3문서를 생성한다:

- **PLAN.md** — 계획서 (plan-manager 스킬의 형식 준수)
- **CONTEXT.md** — 맥락 노트
- **CHECKLIST.md** — 체크리스트 (상태: 🔴 시작 전)

### 4-C: 사용자 검토

```
Phase 4: 초기 개발 계획

첫 번째 기능 "{기능명}" 의 개발 계획을 수립했습니다.

docs/plans/{기능명}/
  ├── PLAN.md       ← 전체 구현 계획
  ├── CONTEXT.md    ← 결정 근거 & 참조 자료
  └── CHECKLIST.md  ← 작업 체크리스트

[계획 요약]
  Phase 1: {단계1 요약}
  Phase 2: {단계2 요약}
  ...

계획을 검토해주세요. 수정할 부분이 있으면 말씀해주세요.
(확인되면 마지막 단계로 넘어갑니다)
```

**여기서 반드시 멈춘다.** 사용자가 "확인", "진행" 등 응답할 때까지 Phase 5로 넘어가지 않는다. 계획에 문제가 있으면 이 단계에서 수정한다.

**절대 금지: Phase 4에서 계획이 승인되었다고 해서 구현을 시작하지 않는다.** Phase 4의 승인은 "계획 내용"에 대한 승인이지 "구현 시작"에 대한 승인이 아니다. 반드시 Phase 5 (환경 세팅)를 먼저 완료해야 한다. `.claude/.initialized` 마커가 생성되기 전에는 어떤 코드도 작성하지 않는다.

## Phase 5: 환경 세팅 적용

> **목표**: 수집한 모든 정보를 바탕으로 설정 파일들을 자동 생성
> **이 Phase를 건너뛰면 Hook, 매뉴얼, CLAUDE.md가 없는 상태로 개발이 시작된다. 반드시 완료해야 한다.**

### 5-A: 생성할 파일 목록

> **모든 파일은 Phase 1에서 `pwd`로 확인한 `PROJECT_ROOT` 절대 경로를 기준으로 생성한다.**
> 예: `PROJECT_ROOT=/Users/alice/projects/myapp` 이면 → `/Users/alice/projects/myapp/CLAUDE.md`

1. **`$PROJECT_ROOT/hooks/config.yml`** — 프로젝트 맞춤 Hook 설정
2. **`$PROJECT_ROOT/skills/dev-manual/chapters/01~06`** — 프로젝트 맞춤 매뉴얼
3. **`$PROJECT_ROOT/CLAUDE.md`** — 프로젝트 전용 CLAUDE.md (아래 템플릿 기반)
4. **추가 에이전트** (Phase 3에서 요청된 경우) — `$PROJECT_ROOT/agents/` 에 추가
5. **`$PROJECT_ROOT/.claude/.initialized`** — 초기화 완료 마커

### 5-B: config.yml 생성 규칙

Phase 2~3의 정보를 기반으로 config.yml을 프로젝트에 맞게 커스터마이징한다:

- **keywords**: 기술 스택 키워드 추가 (React → 컴포넌트/훅/상태 등)
- **intents**: 프로젝트 특성에 맞는 의도 추가 (DB 사용 → migration 의도)
- **locations**: 실제 디렉토리 구조에 맞게 패턴 수정
- **code_patterns**: 기술 스택에 맞는 패턴 추가
- **completion_check.linters**: 실제 사용 린터만 활성화

### 5-C: 매뉴얼 챕터 생성 규칙

| 챕터 | 사용하는 정보 |
|------|---------------|
| 01-project-overview.md | Phase 1 (비전, 기능 목록) + Phase 2 (기술 스택, 구조) |
| 02-coding-standards.md | Phase 3 (코딩 규칙) + Phase 2 (언어/프레임워크별 관례) |
| 03-architecture.md | Phase 2 (구조) + Phase 1 (기능 목록에서 모듈 도출) |
| 04-error-handling.md | Phase 3 (에러 처리) + Phase 2 (프레임워크별 에러 처리) |
| 05-security.md | Phase 3 (보안) + Phase 2 (프레임워크별 보안 가이드) |
| 06-testing.md | Phase 2 (테스트 프레임워크) + Phase 3 (커버리지 기준) |

#### 챕터 작성 규칙

- **구체적으로 작성** — "에러를 잘 처리하세요" (X) → "AppError 클래스를 사용하여 에러 코드와 메시지를 반환" (O)
- **코드 예시 포함** — 해당 프로젝트 기술 스택에 맞는 실제 코드 예시

## 중요: Phase 순서 엄수

이 스킬에서 가장 중요한 것은 **Phase 5까지 반드시 완료하는 것**이다.

```
Phase 1 → (확인) → Phase 2 → (확인) → Phase 3 → (확인) → Phase 4 → (확인) → Phase 5 → 세팅 완료
                                                                                          │
                                                                                    이 이후에만
                                                                                    구현 가능
```

- 각 Phase 사이에 반드시 사용자의 "확인" 또는 "수정 요청"을 기다린다
- 사용자가 수정을 요청하면 해당 Phase 내에서 수정하고 다시 확인을 받는다
- 한 번에 여러 Phase를 한꺼번에 진행하지 않는다
- Phase 5 (파일 생성)는 Phase 1~4가 모두 확인된 후에만 실행한다
- **Phase 4에서 멈추지 않는다.** 계획 3문서 생성은 Phase 4의 산출물일 뿐이다. Phase 5에서 CLAUDE.md, config.yml, 매뉴얼을 생성해야 시스템이 작동한다.
- **`.claude/.initialized` 마커가 없으면 세팅 미완료 상태다.** 이 마커는 Phase 5에서만 생성된다.
